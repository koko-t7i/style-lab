# DESIGN.md — Format Reference

Use this when filling in the prose sections of a DESIGN.md generated by `scripts/extract_design_md.py`.

**Spec lineage.** DESIGN.md originates with **Google Stitch** (`stitch.withgoogle.com/docs/design-md/format/`). The widely-circulated public corpus (`getdesign.md`, `github.com/VoltAgent/awesome-design-md`) follows that Stitch format **with extended sections**. This skill targets that extended canonical so a DESIGN.md emitted here looks the same as the Claude / Stripe / Linear reference files on getdesign.md and is read identically by Stitch, Cursor, and Claude Code.

## What DESIGN.md is and why it matters

A single Markdown file in the repo root that gives **AI design/coding agents** a persistent, structured understanding of a product's visual identity. Every time an agent generates UI, it reads this file first. Two parts:

- **YAML front-matter** — machine-readable tokens (colors, typography, rounded, spacing, components). Agents map these directly to CSS variables / Tailwind config / native theme.
- **Markdown prose** — fixed sections explaining the *why* behind the tokens, so agents make judgment calls the tokens alone can't capture ("when do I use accent vs ink for a CTA?").

Together they let an agent stay on-brand even on prompts the human never anticipated.

`AGENTS.md` tells coding agents *how to build*; `DESIGN.md` tells design agents *how it should look*. They are siblings, not substitutes.

## The 11 sections (canonical order — do not reorder, do not skip)

Headings are plain `## Title` (no numeric prefix) — matching the getdesign.md corpus. Order is fixed.

| # | Section | Sub-headings | What goes here |
|---|---|---|---|
| 1 | **Overview** | (prose) + **Key Characteristics:** bullets | What the design feels like, who it's for, the surface/atmosphere story. End with a `**Key Characteristics:**` bullet list naming the defining color, the display face, the signature visual move, the radius scale, the section rhythm — each citing a `{token}`. **Specific to this variant**; never "modern"/"clean". |
| 2 | **Colors** | `### Brand & Accent` · `### Surface` · `### Text` · `### Semantic` | One bullet per color: bold human name, `{colors.x}` ref, hex, and *what it's for / when to use it*. Contrast strategy and restrictions ("never accent for body text"). If a hex doubles as a gradient stop, say so. |
| 3 | **Typography** | `### Font Family` · `### Hierarchy` · `### Principles` · `### Note on Font Substitutes` | Display vs body face and the editorial logic of the split. A hierarchy table (Token / Size / Weight / Line Height / Letter Spacing / Use). Weight & tracking rules. Open-source substitutes for any licensed/proprietary face. |
| 4 | **Layout** | `### Spacing System` · `### Grid & Container` · `### Whitespace Philosophy` | Base unit + the named spacing tokens, section padding, card padding. Max content width, column grids per content type, responsive column counts. The whitespace rationale. |
| 5 | **Elevation & Depth** | a Level/Treatment/Use table · `### Decorative Depth` | Shadow philosophy as a table (flat / hairline / soft / lifted) with concrete values and when each applies. **State explicitly if depth is rejected** (Brutalism uses hard offset blocks, not blur; flat styles say "no shadow, ever"). |
| 6 | **Shapes** | `### Border Radius Scale` · `### Photography & Illustrations` | The `{rounded.*}` scale and which radius each component uses. What shapes signal interactivity. Image/illustration treatment (crop, stroke weight, art direction). |
| 7 | **Components** | one bold-key entry per component | Per component (`button-primary`, `feature-card`, `top-nav`, `text-input`, `cta-band`, `footer`, …): the token bindings via `{...}` refs and *what it MUST do and MUST NOT do*. The most load-bearing section — agents read it every time they emit a button or card. |
| 8 | **Do's and Don'ts** | `### Do` · `### Don't` | 5–7 each. **Concrete and verifiable** ("16px+ body text", "no gradients", "96px between bands") — never vague ("be consistent"). This is the guardrail list. |
| 9 | **Responsive Behavior** | `### Breakpoints` · `### Touch Targets` · `### Collapsing Strategy` · `### Image Behavior` | A breakpoint table (Name / Width / Key Changes). Minimum touch target sizes per interactive component. How grids/nav collapse mobile→desktop. How code/media behave at small widths. |
| 10 | **Iteration Guide** | numbered list | Rules for safely extending the system: one component at a time, variants live as separate `components:` entries, always `{token.refs}` never inline hex, what is unbreakable (the type split, the surface trinity), how to add emphasis without breaking the brand. |
| 11 | **Known Gaps** | bullets | Honest list of what this document does NOT cover or could not be determined from the source: licensed fonts unavailable as web fonts, animation/transition timings out of scope, states not observable from a static page, product-app surfaces beyond the marketing page. Prevents agents from inventing authority that isn't here. |

## YAML front-matter conventions

```yaml
---
version: alpha               # always "alpha" for now
name: <ProductName>
description: "<one rich sentence: surface, type voice, accent, signature move>"
# Visual direction: <StyleName>

colors:                      # layered & semantic, not just paper/ink/accent
  primary: "#cc785c"         # the signature brand accent
  primary-active: "#a9583e"  # press / hover-darker variant
  primary-disabled: "#e6dfd8"
  ink: "#141413"             # primary text, usually near-black
  body: "#3d3d3a"            # running-text color
  muted: "#6c6a64"           # secondary text, captions
  hairline: "#e6dfd8"        # 1px borders / dividers
  canvas: "#faf9f5"          # default page floor
  surface-soft: "#f5f0e8"    # soft band background
  surface-card: "#efe9de"    # card background, one step off canvas
  surface-dark: "#181715"    # dark product surfaces, footer
  on-primary: "#ffffff"      # text on the accent
  on-dark: "#faf9f5"         # text on dark surfaces
  success: "#5db872"
  warning: "#d4a017"
  error: "#c64545"

gradients:                   # OPTIONAL extension — keep when the brand's
  brand: "linear-gradient(135deg, #5B7FFF 0%, #A78BFA 100%)"  # identity is a gradient
                             # (vanilla Stitch files have no gradients; this
                             #  skill preserves them because many brands' DNA
                             #  IS the gradient. Solids that are also stops get
                             #  renamed brand-start / brand-end in colors:)

typography:                  # FULL named hierarchy, not display/body/mono
  display-xl: { fontFamily: "Copernicus, serif", fontSize: 56px, fontWeight: 400, lineHeight: 1.05, letterSpacing: -1px }
  display-lg: { fontFamily: "Copernicus, serif", fontSize: 44px, fontWeight: 400, lineHeight: 1.1,  letterSpacing: -0.5px }
  display-md: { fontFamily: "Copernicus, serif", fontSize: 32px, fontWeight: 400, lineHeight: 1.15, letterSpacing: -0.3px }
  title-lg:   { fontFamily: "StyreneB, sans-serif", fontSize: 22px, fontWeight: 600, lineHeight: 1.3, letterSpacing: 0 }
  title-md:   { fontFamily: "StyreneB, sans-serif", fontSize: 18px, fontWeight: 600, lineHeight: 1.4, letterSpacing: 0 }
  body-md:    { fontFamily: "StyreneB, sans-serif", fontSize: 16px, fontWeight: 400, lineHeight: 1.6, letterSpacing: 0 }
  body-sm:    { fontFamily: "StyreneB, sans-serif", fontSize: 14px, fontWeight: 400, lineHeight: 1.55, letterSpacing: 0 }
  caption:    { fontFamily: "StyreneB, sans-serif", fontSize: 13px, fontWeight: 500, lineHeight: 1.4, letterSpacing: 0 }
  code:       { fontFamily: "JetBrains Mono, monospace", fontSize: 14px, fontWeight: 400, lineHeight: 1.6, letterSpacing: 0 }
  button:     { fontFamily: "StyreneB, sans-serif", fontSize: 14px, fontWeight: 600, lineHeight: 1, letterSpacing: 0 }
  nav-link:   { fontFamily: "StyreneB, sans-serif", fontSize: 14px, fontWeight: 500, lineHeight: 1.4, letterSpacing: 0 }

rounded:
  xs: 4px
  sm: 6px
  md: 8px
  lg: 12px
  xl: 16px
  pill: 9999px
  full: 9999px

spacing:                     # NAMED scale, not numeric 1..8
  xxs: 4px
  xs: 8px
  sm: 12px
  md: 16px
  lg: 24px
  xl: 32px
  xxl: 48px
  section: 96px

components:                  # tokens referenced via {brace} syntax
  button-primary:
    backgroundColor: "{colors.primary}"
    textColor: "{colors.on-primary}"
    typography: "{typography.button}"
    rounded: "{rounded.md}"
    padding: 12px 20px
  button-secondary:
    backgroundColor: "{colors.canvas}"
    textColor: "{colors.ink}"
    typography: "{typography.button}"
    rounded: "{rounded.md}"
    padding: 12px 20px
  feature-card:
    backgroundColor: "{colors.surface-card}"
    textColor: "{colors.ink}"
    typography: "{typography.title-md}"
    rounded: "{rounded.lg}"
    padding: 32px
  top-nav:
    backgroundColor: "{colors.canvas}"
    textColor: "{colors.ink}"
    typography: "{typography.nav-link}"
    height: 64px
  text-input:
    backgroundColor: "{colors.canvas}"
    textColor: "{colors.ink}"
    typography: "{typography.body-md}"
    rounded: "{rounded.md}"
    padding: 10px 14px
  cta-band:
    backgroundColor: "{colors.primary}"
    textColor: "{colors.on-primary}"
    typography: "{typography.display-md}"
    rounded: "{rounded.lg}"
    padding: 64px
  footer:
    backgroundColor: "{colors.surface-dark}"
    textColor: "{colors.on-dark}"
    typography: "{typography.body-sm}"
    padding: 64px
---
```

**Rules:**
- **Reference tokens with `{brace}` syntax**, not bare dot notation. `rounded: "{rounded.md}"`, `backgroundColor: "{colors.primary}"`, `typography: "{typography.button}"`. Prose cites tokens the same way: ``the cream canvas (`{colors.canvas}` — #faf9f5)``. This matches the getdesign.md corpus and is what Stitch parses.
- **Colors are layered and semantic.** Aim for the role families: brand/accent (`primary`, `primary-active`, `primary-disabled`, `accent-*`), surface (`canvas`, `surface-soft`, `surface-card`, `surface-dark`, …), text (`ink`, `body`, `muted`, `on-primary`, `on-dark`), semantic (`success`, `warning`, `error`). The extractor produces best guesses from luminance/saturation — **rename them to the canonical vocabulary based on actual on-page usage**. A color extracted as `surface` that is actually the footer floor should become `surface-dark`.
- **Typography is a full named hierarchy.** `display-xl … nav-link`, each with `fontFamily / fontSize / fontWeight / lineHeight / letterSpacing`. The extractor seeds families and reasonable defaults; tune sizes/weights/tracking to what the variant actually uses.
- **`spacing` uses the named scale** (`xxs … section`), not arbitrary or numeric keys. `rounded` always includes `pill`/`full` (9999px) even if the variant has no pill — downstream components need the token to exist.
- **Components reference tokens, never hard-code values.** This is the whole point — it lets downstream agents reskin without rewriting components. Add `-active` / `-disabled` / `-focused` as separate `components:` entries when relevant.
- **`gradients:` is a sanctioned extension**, not part of vanilla Stitch. Keep it when the brand's identity *is* a gradient; a solid hex that is also a gradient stop is renamed in `colors:` to `brand-start` / `brand-end` so the relationship is explicit.

## Validation

```bash
npx @google/design.md lint DESIGN.md
```

Catches: broken token references (`{rounded.huge}` when only `xs/sm/md/lg/xl` exist), WCAG contrast failures between text/background pairs, missing required sections, malformed YAML. Fix everything it flags before handoff.

## Common pitfalls when filling in the prose

- **Don't repeat the YAML in prose.** Don't write "the ink color is #141413". The YAML says that. Write the *rationale* — why this ink, when to deviate. (Citing the token+hex inline once for orientation, as the corpus does, is fine; re-listing the whole palette is not.)
- **Don't write generic design-system copy.** "Use whitespace generously" applies to every system; it tells the agent nothing about *this* one. Be specific: "96px between every band; 32px inside cards; the cream-to-dark band alternation IS the pacing mechanism."
- **Ground every claim in the variant page.** If the variant uses a struck-through competitor name in the hero, document that pattern in Components or Do's. A human should be able to find every claim in the index.html.
- **Don't soften opinionated styles.** A Brutalism DESIGN.md must *forbid* gradients, shadows, and rounded corners — not "discourage" them. Downstream agents need hard guardrails, not vibes.
- **Components is the most load-bearing section.** State padding, font weight, hover/press behavior, disabled state — anything that makes the component recognizable. Use `{token}` refs.
- **Fill Responsive / Iteration Guide / Known Gaps for real.** These three are what separate the extended canonical from a toy. Empty or hand-wavy versions are a failed handoff — agents rely on Known Gaps to NOT invent authority, and on Responsive to not ship a desktop-only layout.

## Example: a tight Components entry for a Brutalism brand

> **`button-primary`**
>
> - `backgroundColor: {colors.accent}` on `textColor: {colors.ink}`
> - `border: 3px solid {colors.ink}`
> - `rounded: {rounded.xs}` is wrong here — Brutalism is `border-radius: 0`. This brand should set `rounded.xs: 0px`.
> - `box-shadow: 6px 6px 0 {colors.ink}` — hard offset, no blur. On press: shadow shrinks to `2px 2px 0` and the button translates `(4px, 4px)`.
> - `typography: {typography.button}` with `fontWeight: 900`, `text-transform: uppercase`, `letterSpacing: 0.05em`.
> - **Must not** add a gradient. **Must not** soften the shadow. **Must not** use accent-on-accent (always paired with ink for contrast).

That's the level of specificity that makes DESIGN.md useful instead of decorative.
