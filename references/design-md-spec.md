# DESIGN.md — Format Reference

Use this when filling in the prose sections of a DESIGN.md generated by `scripts/extract_design_md.py`. Spec source: https://github.com/google-labs-code/design.md (Apache-2.0, Google Labs, April 2026).

## What DESIGN.md is and why it matters

A single Markdown file in the repo root that gives **AI coding agents** (Cursor / Claude Code / Stitch / etc.) a persistent, structured understanding of a product's visual identity. Every time an agent generates UI code, it reads this file first. The file has two parts:

- **YAML front-matter** — machine-readable design tokens (colors, typography, spacing, radii, components). Agents map these directly to CSS variables / Tailwind config / native theme.
- **Markdown prose** — 9 fixed sections explaining the *why* behind the tokens. Agents use this to make judgment calls the tokens alone can't capture (e.g. "when do I use accent vs ink for a CTA?").

Together they let an agent stay on-brand even on prompts the human never anticipated.

## The 9 sections (canonical order — do not reorder, do not skip)

| # | Section | What goes here |
|---|---|---|
| 1 | **Overview** | What the design is meant to feel like, who it's for, the tone of voice. 1–2 short paragraphs, **specific to this variant**. Avoid generic adjectives like "modern" or "clean". |
| 2 | **Colors** | Why these hues. What each role means (paper / ink / accent / etc.). Contrast strategy. Restrictions ("never use accent for body text"). Reference token names from front-matter. |
| 3 | **Typography** | Display vs body face. Weight & letter-spacing conventions. Hierarchy rules (h1 size, line-height, paragraph rhythm). Special treatments (drop caps, all-caps tracking, monospace use). |
| 4 | **Layout** | Grid behavior (cols / gutters), max content widths, vertical rhythm, alignment philosophy (left / centered / asymmetric), density (spacious vs dense). Reference spacing tokens. |
| 5 | **Elevation & Depth** | Shadow philosophy — flat? layered? hard offset blocks? glass via `backdrop-filter`? List specific shadow values and when each applies. **State explicitly if depth is avoided** (some styles like Brutalism reject elevation). |
| 6 | **Shapes** | Corner radius philosophy (sharp / rounded / mixed). What shapes signal interactivity. Any unusual shape language (bento tiles, blob backgrounds, ASCII boxes). |
| 7 | **Components** | Key components: primary button, card, nav, form input. For each, say *what it MUST do* and *what it MUST NOT do*. Use token references where possible (e.g. `bg: ink`, `radius: rounded.sm`). |
| 8 | **Do's and Don'ts** | 3–5 specific items each. **Concrete and verifiable** ("16px+ body text", "no gradients") — not vague ("be consistent"). |
| 9 | (Optional extension) | Use only if the brand needs something not covered above (motion guidelines, voice & tone, illustration style, etc.). Skip if not needed. |

## YAML front-matter conventions

```yaml
---
version: alpha               # always "alpha" for now
name: <ProductName>
description: "<one line>"
colors:
  ink: "#0F0E0C"             # primary text color, usually darkest
  paper: "#FAF7F2"           # primary background, usually lightest
  accent: "#B5532E"          # the one saturated brand color
  surface: "#FFFFFF"         # secondary surface (cards, modals)
  muted: "#6B6B6B"           # secondary text
  border: "#E5E5E5"          # dividers, input borders
typography:
  display: { family: "Public Sans", weight: 700, size: "48px" }
  body:    { family: "Public Sans", weight: 400, size: "16px" }
  mono:    { family: "JetBrains Mono", weight: 400, size: "14px" }
spacing:
  1: "4px"
  2: "8px"
  3: "16px"
  4: "24px"
  5: "32px"
  6: "48px"
rounded:
  sm: "4px"
  md: "8px"
  lg: "16px"
components:
  button:
    bg: "accent"
    color: "paper"
    radius: "rounded.sm"
    padding: "spacing.2 spacing.4"
---
```

**Rules:**
- Token names should be **semantic** (`ink`, `accent`), not descriptive (`black`, `red`). Color extraction may produce neutral names like `color01` — rename them based on actual usage.
- Reference other tokens with dot notation: `radius: "rounded.sm"`, not `radius: "4px"`.
- `spacing` keys are typically `1`–`8` representing a scale, not arbitrary values.
- Components should reference tokens, not hard-code values. This is the whole point — it lets downstream agents swap themes without rewriting components.

## Validation

```bash
npx @google/design.md lint DESIGN.md
```

Catches:
- Broken token references (e.g. you wrote `rounded.huge` but only `sm/md/lg` exist)
- WCAG contrast failures between text/background pairs
- Missing required sections
- Malformed YAML

## Common pitfalls when filling in the prose

- **Don't repeat the YAML in prose.** Don't write "the ink color is #0F0E0C". The YAML already says that. Write the *rationale* — why ink, when to deviate.
- **Don't write generic design-system copy.** "Use whitespace generously" applies to every design system; it tells the agent nothing about *this* one. Be specific: "Maintain at least 96px of vertical space between the hero and the first feature row."
- **Reference what's actually on the variant page.** If the variant uses a struck-through "Datadog" in the hero as a positioning move, mention that pattern in Components or Do's. Ground every claim in something a human could find in the index.html.
- **Don't soften opinionated styles.** A Brutalism DESIGN.md should *forbid* gradients, shadows, and rounded corners — not merely "discourage" them. The whole point is downstream agents need clear guardrails.
- **Components section is the most load-bearing.** This is what agents will read every time they generate a button or a card. Be precise. State padding, font weight, hover behavior, disabled state — anything that makes the component recognizable.

## Example: a tight Components section for a Brutalism brand

> ### Primary button
>
> - `bg: accent` on `color: ink`
> - `border: 3px solid ink`
> - `border-radius: 0` — never rounded.
> - `box-shadow: 6px 6px 0 ink` — hard offset, no blur. On hover: shadow shrinks to `2px 2px 0 ink` and button translates `(4px, 4px)`.
> - `font-family: typography.display`, `font-weight: 900`, `text-transform: uppercase`, `letter-spacing: 0.05em`.
> - **Don't** add a gradient. **Don't** soften the shadow. **Don't** use accent on accent (always paired with ink for contrast).

That's the level of specificity that makes DESIGN.md useful instead of decorative.
